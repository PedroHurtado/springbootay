# Módulo 4 · Lección 4.7 — Despacho de eventos en la transacción de persistencia

> **Duración estimada:** 50 min
> **Servicio del proyecto:** `shopmicro-infrastructure` (el repositorio / persistencia)
> **Requisitos:** Lección 4.6 (el flujo de emisión), DI de Spring y PipelinR

---

## 1. Dónde enganchamos el despacho

Tenemos agregados que acumulan eventos en memoria. La pregunta es: **¿en qué momento exacto se
publican esos eventos?**

La respuesta de ShopMicro: **justo al persistir el agregado, dentro de la misma transacción y
antes del commit.** En Java/JPA no existe un único `SaveChanges` global como en otros ecosistemas:
el equivalente natural es **personalizar el `save` del repositorio** (la base de Spring Data) para
que, al guardar una raíz de agregado:

1. Recoja los eventos pendientes del agregado.
2. Los publique uno a uno con PipelinR.
3. Limpie los eventos del agregado.
4. Delegue en el `save` real de JPA —que se confirmará al cerrar la transacción `@Transactional`.

> **¿Por qué personalizar el repositorio base y no un *listener*?** Un
> `@TransactionalEventListener(phase = BEFORE_COMMIT)` o un *interceptor* de Hibernate son
> perfectamente válidos y, en producción, a menudo preferibles (desacoplan la lógica del
> repositorio). Pero para **enseñar el flujo** el `save` personalizado es más directo: todo el
> mecanismo está a la vista en un solo método. Cuando el concepto está claro, migrar a un listener
> es trivial.

## 2. El repositorio base paso a paso

Spring Data permite sustituir la implementación base de **todos** los repositorios por una
propia. Ahí interceptamos el `save`:

```java
package com.shopmicro.infrastructure.persistence;

import an.awesome.pipelinr.Pipeline;
import com.shopmicro.domain.AggregateRoot;
import com.shopmicro.domain.DomainEvent;
import jakarta.persistence.EntityManager;
import org.springframework.data.jpa.repository.support.JpaEntityInformation;
import org.springframework.data.jpa.repository.support.SimpleJpaRepository;
import org.springframework.transaction.annotation.Transactional;

import java.util.List;

// Repositorio base para raíces de agregado: despacha eventos antes de persistir.
public class EventPublishingRepository<T extends AggregateRoot<?, ?>, ID>
        extends SimpleJpaRepository<T, ID> {

    private final Pipeline pipeline;

    public EventPublishingRepository(JpaEntityInformation<T, ?> entityInformation,
                                     EntityManager entityManager,
                                     Pipeline pipeline) {
        super(entityInformation, entityManager);
        this.pipeline = pipeline;
    }

    @Override
    @Transactional
    public <S extends T> S save(S aggregate) {
        dispatchDomainEvents(aggregate);
        return super.save(aggregate);
    }

    private void dispatchDomainEvents(AggregateRoot<?, ?> aggregate) {
        // 1. Materializar los eventos ANTES de publicar (ver nota más abajo).
        List<? extends DomainEvent<?>> domainEvents = List.copyOf(aggregate.getDomainEvents());

        // 2. Limpiar ANTES de publicar: si un handler vuelve a tocar el agregado,
        //    no reprocesamos los mismos eventos.
        aggregate.clear();

        // 3. Publicar cada evento con PipelinR. Los handlers se ejecutan aquí,
        //    dentro de la misma transacción, antes del commit.
        domainEvents.forEach(pipeline::send);
    }
}
```

Para activarlo como base de todos los repositorios JPA:

```java
package com.shopmicro.infrastructure.persistence;

import org.springframework.context.annotation.Configuration;
import org.springframework.data.jpa.repository.config.EnableJpaRepositories;

@Configuration
@EnableJpaRepositories(
        basePackages = "com.shopmicro",
        repositoryBaseClass = EventPublishingRepository.class)
public class PersistenceConfig {
}
```

> **Matiz frente a otros ORMs.** Aquí el barrido es **por raíz de agregado guardada**
> (`repository.save(agg)`), que es el uso idiomático en Spring Data: tú guardas la raíz. No hay
> un *change tracker* global que descubra solo todos los agregados modificados; eres tú quien
> guarda explícitamente cada raíz. Es una diferencia honesta de modelo, y normalmente la deseable:
> el límite del agregado se respeta de forma natural.

## 3. Los dos detalles que SÍ importan

### 3.1 Copia defensiva: la colección que muta mientras la recorres

Publicamos **antes** del `super.save()`. Eso significa que un handler del Camino 1 hará
`outboxRepository.save(...)` mientras seguimos dentro del bloque de despacho. Si estuviéramos
iterando directamente sobre la lista de eventos del agregado y un handler modificara esa colección
(o el propio agregado), tendríamos un `ConcurrentModificationException`.

Por eso materializamos con `List.copyOf(...)` **antes** de empezar a publicar: congelamos la lista
de eventos de *esta* operación. Las nuevas filas del Outbox que añadan los handlers se persistirán
en el commit final de la transacción, pero no entran en el bucle de despacho actual.

### 3.2 `clear()` antes de publicar, no después

Limpiamos los eventos del agregado **antes** del bucle de `send`. ¿Por qué? Porque un handler
podría, legítimamente, modificar el mismo agregado y volver a guardarlo en cadena. Si no hubiéramos
limpiado, esos eventos seguirían en la colección y se republicarían. Limpiar primero corta
cualquier ciclo de reprocesamiento.

> Como ya **materializamos los eventos en `domainEvents`** (paso 1), limpiar la colección del
> agregado (paso 2) no nos quita nada: seguimos teniendo la lista para publicar.

## 4. El contrato `HasDomainEvents`

Si quieres que el despacho pueda preguntar "¿este objeto tiene eventos?" sin conocer los
parámetros genéricos `<ID, EID>`, expón un contrato no genérico. Esto es **ISP**: el despachador
solo necesita *leer eventos y limpiarlos*, nada más.

```java
package com.shopmicro.domain;

import java.util.List;

public interface HasDomainEvents {
    List<? extends DomainEvent<?>> domainEvents();
    void clear();
}
```

Y la `AggregateRoot` lo implementa, adaptando su colección genérica al contrato no genérico:

```java
public abstract class AggregateRoot<ID, EID>
        extends EntityBase<ID> implements HasDomainEvents {

    private final List<DomainEvent<EID>> domainEvents = new ArrayList<>();

    // ... add / remove / clear como en 4.6 ...

    @Override
    public List<? extends DomainEvent<?>> domainEvents() {
        return Collections.unmodifiableList(domainEvents);
    }
}
```

> **Detalle sobre PipelinR y el despacho por tipo.** PipelinR despacha según el tipo del objeto
> publicado: localiza el/los `Notification.Handler` registrados para el tipo concreto de cada
> evento. Para que un `DomainEvent<ID>` sea publicable debe implementar la interfaz `Notification`
> de PipelinR (lo añadimos en la lección 4.8, donde definimos los handlers).

---

## 5. Por qué esto garantiza atomicidad

El orden es deliberado:

```
dispatchDomainEvents()   →  los handlers escriben vía EntityManager
                            (filas de Outbox, efectos del Camino 2)
                            PERO TODAVÍA NO HAY COMMIT
super.save(aggregate)    →  persiste el estado del agregado
cierre de @Transactional →  AQUÍ se hace el ÚNICO commit:
                            estado del agregado + filas de Outbox + Camino 2
                            en una sola transacción de base de datos
```

Si la transacción falla al confirmar, **nada** se persiste: ni el cambio de estado, ni las filas
del Outbox, ni el efecto local. No existe el escenario "guardé el evento pero no el cambio que lo
provocó". Esa es justamente la garantía que el patrón Outbox necesita, y la obtenemos por poner el
despacho **dentro de la misma transacción**, antes del commit.

---

## Puntos clave

- El despacho se engancha **personalizando el `save` del repositorio base** de Spring Data:
  materializar → limpiar → publicar (PipelinR) → `super.save`. (Un `@TransactionalEventListener`
  o un interceptor de Hibernate es la evolución natural, pero el `save` personalizado enseña
  mejor.)
- **Copia defensiva con `List.copyOf` antes de publicar**: los handlers del Camino 1 añaden filas
  vía `EntityManager` y podrían modificar la colección que estás recorriendo.
- **`clear()` antes del bucle de `send`** para cortar reprocesamientos en cadena; los eventos ya
  están a salvo en una lista materializada.
- El despachador depende de un contrato **`HasDomainEvents` no genérico** (ISP): solo necesita
  leer y limpiar eventos.
- Publicar **dentro de la transacción**, antes del commit, mete los efectos de los handlers en la
  **misma transacción** → **un único commit atómico** al cerrar `@Transactional`.
