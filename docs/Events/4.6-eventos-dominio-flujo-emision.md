# Módulo 4 · Lección 4.6 — Eventos de dominio: el flujo de emisión

> **Duración estimada:** 45 min
> **Servicio del proyecto:** transversal (la infraestructura vive en `shopmicro-domain` + `shopmicro-infrastructure`)
> **Requisitos:** haber visto mensajería asíncrona (conceptos: *broker*, cola, *topic*, eventos) en 4.1

---

## 1. Qué problema resolvemos

Cuando un agregado cambia de estado, a menudo *otras partes del sistema* necesitan enterarse:
otro caso de uso del mismo servicio, o incluso otro microservicio. La forma ingenua —llamar
directamente a esas otras partes desde el agregado o desde el handler del comando— acopla el
agregado a sus consumidores y mezcla responsabilidades.

Un **evento de dominio** invierte esa relación: el agregado se limita a **declarar que algo ha
ocurrido** ("el precio del producto ha cambiado", "se ha creado un pedido") y *no sabe quién
reaccionará*. Quien reacciona son los **EventHandlers**, registrados aparte.

La pregunta de esta lección no es *qué* es un evento de dominio (eso ya lo modelamos), sino
**cómo se emite**: desde que el agregado lo acumula hasta que alguien lo procesa.

> **El mediador: PipelinR.** Para despachar comandos y eventos usamos **PipelinR**, una librería
> ligera de Java que implementa el patrón mediador (comando/handler y notificación/handlers).
> Es el equivalente conceptual de un *mediator* clásico: el emisor publica; uno o varios
> handlers reaccionan, sin conocerse entre sí.

## 2. Las piezas que ya tenemos

Partimos de dos clases que ya están definidas en el dominio de ShopMicro.

### 2.1 El evento de dominio

`DomainEvent<ID>` es **a su vez una entidad** (tiene identidad propia). El identificador es
genérico a propósito: cuando el evento se persista en un *outbox*, cada microservicio elige el
tipo de id que necesite (no se asume `UUID`). El `payload` es `Object`: puede ser un `record`,
una clase, o el propio agregado.

```java
package com.shopmicro.domain;

public enum DomainEventStatus {
    PENDING,
    PUBLISHED
}
```

```java
package com.shopmicro.domain;

import java.time.Instant;
import java.util.UUID;

public class DomainEvent<ID> extends EntityBase<ID> {

    private final String eventType;
    private final String aggregate;
    private final UUID aggregateId;
    private final String user;
    private final Instant timestamp;
    private DomainEventStatus status;
    private final Object payload;

    protected DomainEvent(ID id, String eventType, String aggregate, UUID aggregateId,
                          String user, Instant timestamp, Object payload) {
        super(id);
        this.eventType = eventType;
        this.aggregate = aggregate;
        this.aggregateId = aggregateId;
        this.user = user;
        this.timestamp = timestamp;
        this.status = DomainEventStatus.PENDING;
        this.payload = payload;
    }

    public static <ID> DomainEvent<ID> create(ID id, String eventType, String aggregate,
                                              UUID aggregateId, String user,
                                              Instant timestamp, Object payload) {
        return new DomainEvent<>(id, eventType, aggregate, aggregateId, user, timestamp, payload);
    }

    public void markAsPublished() {
        this.status = DomainEventStatus.PUBLISHED;
    }

    public String getEventType()     { return eventType; }
    public String getAggregate()     { return aggregate; }
    public UUID getAggregateId()     { return aggregateId; }
    public String getUser()          { return user; }
    public Instant getTimestamp()    { return timestamp; }
    public DomainEventStatus getStatus() { return status; }
    public Object getPayload()       { return payload; }
}
```

> **Detalle de diseño.** Constructor `protected` y atributos `final` expuestos solo por
> *getter*: el evento es **inmutable** una vez creado. Lo único que muta es el `status`, y solo
> a través de `markAsPublished()`. Un evento es un hecho del pasado; un hecho no se reescribe.

### 2.2 La raíz de agregado

La `AggregateRoot` **acumula** los eventos que va generando. La colección se expone como solo
lectura: nadie de fuera puede manipular la lista directamente.

```java
package com.shopmicro.domain;

import java.util.ArrayList;
import java.util.Collections;
import java.util.List;

public abstract class AggregateRoot<ID, EID> extends EntityBase<ID> {

    private final List<DomainEvent<EID>> domainEvents = new ArrayList<>();

    protected AggregateRoot(ID id) {
        super(id);
    }

    public List<DomainEvent<EID>> getDomainEvents() {
        return Collections.unmodifiableList(domainEvents);
    }

    protected void add(DomainEvent<EID> domainEvent) {
        domainEvents.add(domainEvent);
    }

    protected void remove(DomainEvent<EID> domainEvent) {
        domainEvents.remove(domainEvent);
    }

    public void clear() {
        domainEvents.clear();
    }
}
```

> **Por qué dos parámetros de tipo.** `ID` es el id del agregado; `EID` es el id de los eventos.
> Son independientes porque **el outbox decide con qué tipo de id persiste cada evento**, y eso
> no tiene por qué coincidir con el id del agregado.

## 3. El flujo completo, de un vistazo

```
  Comando (Handler de la feature, despachado con PipelinR)
        │  agg.hacerAlgo()  → internamente agg.add(evento)
        ▼
  El agregado acumula DomainEvents en memoria
        │
        ▼
  repository.save(agg)  dentro de un método @Transactional
        │
        ├─ 1. Recoge el/los agregado(s) que tienen DomainEvents
        ├─ 2. Publica cada evento con PipelinR (pipeline.send) ───┐
        │                                                          │
        │      PipelinR despacha a 1..N Notification.Handler       │
        │                                                          │
        │   ┌────────────────────────────┬─────────────────────────┴─────┐
        │   │ CAMINO 1 (a otro servicio) │ CAMINO 2 (efecto local)        │
        │   │ el handler hace            │ el handler procesa el efecto   │
        │   │ outboxRepository.save(e)   │ sobre el MISMO EntityManager    │
        │   │ → fila Outbox (PENDING)    │ (cálculo, otra escritura...)   │
        │   └────────────────────────────┴────────────────────────────────┘
        │
        ├─ 3. clear() de los eventos en cada agregado
        └─ 4. el commit ocurre al cerrar la transacción  ← UN ÚNICO COMMIT atómico
                                          (estado + outbox + camino 2)
        │
        ▼  (proceso aparte, en segundo plano)
  MessageRelay (@Scheduled)
        ├─ lee Outbox WHERE status = PENDING
        ├─ publica el mensaje al broker (RabbitMQ)
        └─ markAsPublished()  (PENDING → PUBLISHED)
```

La idea que **debe quedar grabada**: el evento de dominio **no sabe cuál es su destino**. Esa
política la deciden los handlers. Por eso hay dos caminos posibles y ambos parten del mismo
punto.

---

## 4. Los dos caminos

Un mismo evento puede tener varios handlers, y cada handler elige qué hacer con él. En ShopMicro
distinguimos dos destinos:

| | **Camino 1 — Outbox → broker** | **Camino 2 — efecto local** |
|---|---|---|
| **Cuándo** | el efecto cruza la frontera del servicio (otro microservicio debe enterarse) | el efecto se queda dentro del servicio |
| **Qué hace el handler** | inserta el `DomainEvent` en la tabla Outbox | ejecuta el efecto sobre el mismo `EntityManager` |
| **Quién lo publica luego** | el `MessageRelay`, en segundo plano | nadie: ya está hecho al cerrar la transacción |
| **Garantía** | el evento no se pierde aunque el broker esté caído | atómico con el cambio de estado |

> **Punto innegociable del diseño.** Tanto la escritura en el Outbox (Camino 1) como el efecto
> local (Camino 2) ocurren **en la misma transacción** que el cambio de estado del agregado. Hay
> **un solo commit** al final (el de `@Transactional`). Si algo falla, *todo* se revierte: el
> estado, la fila del outbox y el efecto local. No hay estados intermedios incoherentes.

Esto tiene una consecuencia que conviene decir en voz alta: **los handlers del Camino 2 no deben
tener efectos externos no transaccionales** (llamar por HTTP a otro servicio, enviar un correo)
ejecutados directamente. Si el efecto cruza la frontera del proceso, su sitio es el Camino 1
(Outbox). El Camino 2 es para escrituras sobre la propia base de datos y cálculos en memoria. Si
más adelante necesitamos coordinar una operación entre servicios con compensaciones, eso es un
**Saga** (Módulo 6), no un handler in-process.

## 5. Lo que construiremos en las próximas lecciones

Este flujo se reparte en dos piezas de infraestructura:

1. **4.7 — El despacho en la transacción.** Personalizamos el `save` del repositorio (o un
   *Unit of Work*) para detectar agregados con eventos, publicarlos con PipelinR y limpiarlos,
   todo dentro de la transacción y antes del commit. Aquí viven los dos caminos.

2. **4.8 — Los EventHandlers y el MessageRelay.** Escribimos un handler del Camino 1 (al
   Outbox), un handler del Camino 2 (efecto local) y el componente `@Scheduled` que drena el
   Outbox hacia RabbitMQ.

## Puntos clave

- Un **evento de dominio** es un *hecho del pasado*: inmutable, con identidad propia. El agregado
  lo **acumula**; no lo procesa.
- El agregado **no conoce a sus consumidores**. La política de "qué hacer con el evento" vive en
  los **EventHandlers** (handlers de PipelinR), no en el dominio.
- Hay **dos caminos** desde un evento: salir del servicio vía **Outbox → MessageRelay → broker**
  (Camino 1), o resolverse **localmente** sobre el mismo `EntityManager` (Camino 2).
- Todo —cambio de estado, escritura en Outbox y efecto local— ocurre en **una sola transacción**
  con **un único commit**. Atomicidad total o nada.
- Los efectos que **cruzan la frontera del servicio** van siempre por el Outbox, nunca como
  efecto in-process directo.
