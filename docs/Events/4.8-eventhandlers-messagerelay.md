# Módulo 4 · Lección 4.8 — EventHandlers y MessageRelay

> **Duración estimada:** 55 min
> **Servicio del proyecto:** `shopmicro-infrastructure` (handlers + relay) · `ordering-service` (ejemplo `OrderCreated`)
> **Requisitos:** Lección 4.7 (despacho en la transacción), PipelinR, conceptos de broker / RabbitMQ

---

## 1. Cierre del flujo

En 4.7 dejamos el repositorio publicando cada `DomainEvent` con PipelinR justo antes del commit.
Ahora escribimos **quién escucha** esos eventos (los `Notification.Handler`) y, para el Camino 1,
**quién drena el Outbox hacia el broker** (el `MessageRelay`).

Recordemos los dos caminos de la lección 4.6:

```
DomainEvent publicado con PipelinR (pipeline.send)
   ├── Camino 1: handler → escribe en Outbox → (MessageRelay) → RabbitMQ
   └── Camino 2: handler → efecto local sobre el mismo EntityManager
```

---

## 2. El evento como `Notification`

Para que PipelinR pueda despachar un `DomainEvent<ID>`, este debe implementar `Notification`. Lo
añadimos a la clase base; así **todos** los eventos de dominio son publicables sin tocar cada uno.

```java
package com.shopmicro.domain;

import an.awesome.pipelinr.Notification;

public class DomainEvent<ID> extends EntityBase<ID> implements Notification {
    // ... resto idéntico a la lección 4.6 ...
}
```

> **Nota de capas.** Esto introduce una dependencia de PipelinR (`an.awesome.pipelinr`) en el
> módulo de dominio. `Notification` es solo un *marker interface* sin comportamiento. Si te
> incomoda tocar el dominio, la alternativa es envolver el `DomainEvent` en un
> `record NotificationWrapper(Object event) implements Notification` en infraestructura y publicar
> el wrapper. En el curso elegimos el marker directo por simplicidad; lo señalamos para que la
> decisión sea consciente.

---

## 3. Camino 1 — el handler que escribe en el Outbox

Como el `DomainEvent<ID>` **es** la entidad del Outbox (se mapea directamente a la tabla), este
handler no transforma nada: simplemente lo registra. Al estar dentro del despacho del `save` (la
transacción), esa inserción entra en la **misma transacción**.

```java
package com.shopmicro.infrastructure.eventhandlers;

import an.awesome.pipelinr.Notification;
import com.shopmicro.infrastructure.outbox.OutboxRepository;
import org.springframework.stereotype.Component;

// Handler para los eventos que deben SALIR del servicio.
// Recibe el evento concreto (aquí, el de Ordering).
@Component
public class OrderCreatedToOutboxHandler implements Notification.Handler<OrderCreated> {

    private final OutboxRepository outboxRepository;

    public OrderCreatedToOutboxHandler(OutboxRepository outboxRepository) {
        this.outboxRepository = outboxRepository;
    }

    @Override
    public void handle(OrderCreated notification) {
        // El evento ES la fila del Outbox: lo guardamos tal cual.
        // No hay commit aquí; lo hará el cierre de @Transactional de la lección 4.7.
        outboxRepository.save(notification);
    }
}
```

> **Mapeo JPA del Outbox.** La tabla Outbox guarda los campos del `DomainEvent` (`eventType`,
> `aggregate`, `aggregateId`, `user`, `timestamp`, `status`) y el `payload` serializado a JSON
> (`jsonb` en PostgreSQL). El `status` arranca en `PENDING`. La conversión del `payload` se
> configura con `@JdbcTypeCode(SqlTypes.JSON)` (Hibernate 6) o un `AttributeConverter`; lo
> detallamos en el Módulo 6 (JPA con PostgreSQL), aquí basta con saber que la fila queda
> persistida.

## 4. Camino 2 — el handler de efecto local

Este handler resuelve el efecto **dentro del servicio**, sobre el mismo `EntityManager`. No toca
el Outbox ni el broker. Ejemplo: cuando se crea un pedido, marcar como reservado algún contador
interno (efecto que vive en la misma base de datos del servicio de pedidos).

```java
package com.shopmicro.infrastructure.eventhandlers;

import an.awesome.pipelinr.Notification;
import org.springframework.stereotype.Component;

@Component
public class OrderCreatedLocalHandler implements Notification.Handler<OrderCreated> {

    @Override
    public void handle(OrderCreated notification) {
        // Efecto sobre el MISMO EntityManager. Entra en la misma transacción.
        // Sin llamadas HTTP ni envíos de correo: si el efecto cruza la
        // frontera del servicio, su sitio es el Outbox (Camino 1), no aquí.
        // ... lógica local sobre el modelo del servicio ...
    }
}
```

> **Un evento, varios handlers.** `OrderCreated` tiene **dos** handlers registrados: el del Outbox
> y el local. PipelinR los ejecuta a ambos cuando el repositorio publica el evento. Cada uno decide
> su destino sin que el otro lo sepa. Ahí está la potencia del patrón: la política de reacción está
> **distribuida en handlers**, no concentrada en el agregado.

## 5. El MessageRelay: drenar el Outbox hacia RabbitMQ

Hasta aquí, el Camino 1 solo ha *guardado* el evento en una tabla. Falta **publicarlo al broker**.
Eso lo hace un componente `@Scheduled` que se ejecuta en segundo plano, independientemente de la
petición que originó el evento.

```java
package com.shopmicro.infrastructure.messaging;

import com.shopmicro.domain.DomainEventStatus;
import com.shopmicro.infrastructure.outbox.OutboxRepository;
import org.springframework.scheduling.annotation.Scheduled;
import org.springframework.stereotype.Component;
import org.springframework.transaction.annotation.Transactional;

import java.util.List;

@Component
public class MessageRelay {

    private final OutboxRepository outboxRepository;
    private final EventBus eventBus;

    public MessageRelay(OutboxRepository outboxRepository, EventBus eventBus) {
        this.outboxRepository = outboxRepository;
        this.eventBus = eventBus;
    }

    @Scheduled(fixedDelay = 5000)   // cada 5 s
    @Transactional
    public void drainOutbox() {
        // 1. Leer los pendientes (orden de inserción → orden de publicación).
        var pending = outboxRepository
                .findByStatusOrderByTimestampAsc(DomainEventStatus.PENDING, 50);

        if (pending.isEmpty()) {
            return;
        }

        // 2. Publicar cada uno al broker y marcarlo como publicado.
        for (var domainEvent : pending) {
            eventBus.publish(domainEvent);
            domainEvent.markAsPublished();   // PENDING → PUBLISHED
        }

        // 3. El commit de @Transactional persiste los cambios de status.
    }
}
```

`EventBus` es la abstracción del broker; su implementación con RabbitMQ (Spring AMQP) usa
`RabbitTemplate`. Aquí nos basta su contrato:

```java
package com.shopmicro.infrastructure.messaging;

public interface EventBus {
    void publish(Object event);
}
```

```java
package com.shopmicro.infrastructure.messaging;

import org.springframework.amqp.rabbit.core.RabbitTemplate;
import org.springframework.stereotype.Component;

@Component
public class RabbitMqEventBus implements EventBus {

    private final RabbitTemplate rabbitTemplate;

    public RabbitMqEventBus(RabbitTemplate rabbitTemplate) {
        this.rabbitTemplate = rabbitTemplate;
    }

    @Override
    public void publish(Object event) {
        rabbitTemplate.convertAndSend("shopmicro.events", event.getClass().getSimpleName(), event);
    }
}
```

> **El scheduler necesita su contexto.** `@Scheduled` se ejecuta en un hilo del *task scheduler*,
> aparte de cualquier petición. Marcamos el método `@Transactional` para que la lectura del Outbox
> y la actualización del `status` ocurran en su propia transacción. Habilita el scheduling con
> `@EnableScheduling` en una clase de configuración.

## 6. Garantías y límites del relay

Conviene ser honesto con los alumnos sobre qué garantiza este relay sencillo y qué no:

- **At-least-once, no exactly-once.** Si el proceso muere *después* de `publish` pero *antes* del
  commit, ese evento se volverá a publicar en la próxima vuelta. El consumidor debe ser
  **idempotente**. Es el comportamiento estándar y aceptable de un Outbox; lo exactly-once no
  existe en sistemas distribuidos sin coste enorme.
- **Orden aproximado.** Ordenamos por `timestamp`, suficiente para el curso. El orden estricto
  entre eventos de distintos agregados no está garantizado y rara vez se necesita.
- **Polling, no push.** Sondeamos cada 5s. Para producción se puede bajar el intervalo o combinar
  con notificaciones, pero el polling es robusto y fácil de razonar.

> **Por qué un proceso aparte y no publicar al broker dentro de la transacción.** Si publicáramos
> a RabbitMQ dentro de la transacción y el broker estuviera caído, fallaría toda la operación
> —perderíamos también el cambio de estado. Separar la escritura (transaccional, en la DB del
> servicio) de la publicación (en segundo plano, reintentable) es **exactamente** lo que el patrón
> Outbox resuelve.

## 7. Registro en el contenedor de Spring

```java
package com.shopmicro.infrastructure.config;

import an.awesome.pipelinr.Command;
import an.awesome.pipelinr.Notification;
import an.awesome.pipelinr.Pipeline;
import an.awesome.pipelinr.Pipelinr;
import org.springframework.beans.factory.ObjectProvider;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.scheduling.annotation.EnableScheduling;

@Configuration
@EnableScheduling
public class PipelinrConfig {

    // PipelinR recoge TODOS los handlers (comandos y notificaciones) del contenedor.
    @Bean
    Pipeline pipeline(ObjectProvider<Command.Handler> commandHandlers,
                      ObjectProvider<Notification.Handler> notificationHandlers) {
        return new Pipelinr()
                .with(commandHandlers::stream)
                .with(notificationHandlers::stream);
    }
}
```

Los `@Component` `OrderCreatedToOutboxHandler` y `OrderCreatedLocalHandler` se registran solos como
beans; PipelinR los descubre por implementar `Notification.Handler`. RabbitMQ se configura con el
*starter* `spring-boot-starter-amqp` y las propiedades `spring.rabbitmq.*` en `application.yml`.

---

## Puntos clave

- El `DomainEvent<ID>` implementa **`Notification`** para que PipelinR lo despache; un marker
  interface sin comportamiento.
- **Camino 1**: el handler guarda el evento en el repositorio del Outbox —como el evento *es* la
  fila, no hay mapeo— y entra en la **misma transacción** del `save`.
- **Camino 2**: el handler resuelve el efecto sobre el mismo `EntityManager`; nada que cruce la
  frontera del servicio.
- **Un evento, varios handlers**: PipelinR ejecuta todos; la política de reacción está repartida
  en los handlers, no en el agregado.
- El **`MessageRelay`** es un componente **`@Scheduled` + `@Transactional`** que sondea el Outbox,
  publica los `PENDING` al broker y los marca `PUBLISHED`.
- Garantía **at-least-once** → los consumidores deben ser **idempotentes**. Separar escritura de
  publicación es la razón de ser del Outbox.
