# Módulo 4 · Lección 4.9 — Alternativa: `@TransactionalEventListener`

> **Duración estimada:** 30 min
> **Servicio del proyecto:** transversal (`core`)
> **Requisitos:** Lección 4.7 (despacho en la transacción) y la implementación del proyecto
> (convergencia en `RepositoryJpa#save` + PipelinR)

---

## 1. Por qué esta lección

En el proyecto hemos despachado los eventos de dominio **dentro del `save` del repositorio**
(el punto de convergencia de `add`/`update`), publicándolos con PipelinR antes de mapear a JPA.
Funciona y deja todo el mecanismo a la vista en un solo método.

Pero no es la única forma, ni la que más se ve en producción. La alternativa clásica en Spring
es **publicar los eventos como `ApplicationEvent` y escucharlos con
`@TransactionalEventListener`**. Esta lección la explica para que la conozcáis y entendáis los
*trade-offs*: no la implementamos en el código del curso, es material de lectura.

---

## 2. La idea

Spring tiene su propio bus de eventos en proceso: `ApplicationEventPublisher` publica,
y los métodos anotados con `@EventListener` (o `@TransactionalEventListener`) reaccionan.

`@TransactionalEventListener` añade algo que un `@EventListener` normal no tiene: **se engancha a
una fase de la transacción en curso**. Puede ejecutarse:

- `BEFORE_COMMIT` — justo **antes** del commit (todavía dentro de la transacción).
- `AFTER_COMMIT` — después del commit (fase por defecto).
- `AFTER_ROLLBACK` / `AFTER_COMPLETION`.

Para el patrón Outbox nos interesa **`BEFORE_COMMIT`**: la fila del Outbox debe escribirse en la
**misma transacción** que el cambio de estado del agregado (igual que en 4.7). Si usáramos
`AFTER_COMMIT`, el estado ya estaría confirmado y la escritura del Outbox iría en otra transacción
—rompería la atomicidad que el Outbox necesita.

```
  Handler de comando  (@Transactional)
        │  pizza.create()  → el agregado acumula DomainEvents
        │  repository.save(pizza)
        │       ├─ publisher.publishEvent(domainEvent)   ← en vez de pipeline.send(...)
        │       └─ jpa().save(mapper.toJpa(pizza))
        ▼
  @TransactionalEventListener(phase = BEFORE_COMMIT)
        └─ outboxRepository.save(...)   ← misma transacción, antes del commit
        ▼
  commit único  (estado del agregado + fila del Outbox)
```

---

## 3. Cómo quedaría en el proyecto

### 3.1 Recoger y publicar en el punto de convergencia

El **lugar de emisión no cambia**: sigue siendo el `save` del repositorio (donde el agregado
todavía tiene sus eventos, antes del mapper). Lo único que cambia es **a quién** publicamos: en
lugar de `pipeline().send(event)`, usamos el `ApplicationEventPublisher` de Spring.

```java
public interface RepositoryJpa<T extends EntityBase, ID, J> {
    JpaRepository<J, ID> jpa();
    Mapper<T, J> mapper();
    ApplicationEventPublisher publisher();   // en vez de Pipeline pipeline()

    default J save(T entity) {
        dispatchDomainEvents(entity);
        return jpa().save(mapper().toJpa(entity));
    }

    private void dispatchDomainEvents(T entity) {
        if (entity instanceof AggregateBase aggregate) {
            var events = List.copyOf(aggregate.getEvents());
            aggregate.clearEvents();
            events.forEach(publisher()::publishEvent);   // Spring, no PipelinR
        }
    }
}
```

> Fíjate en que **el paso de recolectar los eventos del agregado sigue existiendo**. El listener
> resuelve *quién reacciona*, pero no de dónde salen los eventos: eso lo hace igualmente el
> write-path. Por eso, en el dilema de "¿dónde lanzo los eventos?", el listener no elimina el
> punto de convergencia; lo complementa.

### 3.2 El listener que escribe en el Outbox

```java
@Component
public class DomainEventToOutboxListener {

    private final OutboxRepository outboxRepository;
    private final ObjectMapper objectMapper;

    public DomainEventToOutboxListener(OutboxRepository outboxRepository, ObjectMapper objectMapper) {
        this.outboxRepository = outboxRepository;
        this.objectMapper = objectMapper;
    }

    @TransactionalEventListener(phase = TransactionPhase.BEFORE_COMMIT)
    public void on(DomainEvent event) throws Exception {
        outboxRepository.save(new Outbox(
                event.type(), event.occurredOn(), event.domainType(), event.domainId(),
                /* user */ null, objectMapper.writeValueAsString(event.payload()), /* traceId */ null));
    }
}
```

Comparado con el `Notification.Handler<DomainEvent>` de PipelinR, el cuerpo es prácticamente
idéntico: cambia la anotación de enganche y la forma de descubrir el handler.

---

## 4. Detalles que SÍ importan

- **Solo se dispara si hay transacción.** `@TransactionalEventListener` por defecto **ignora** el
  evento si no hay una transacción activa (`fallbackExecution = false`). Como publicamos dentro
  del `@Transactional` del handler de comando, hay transacción y el listener corre. Si publicaras
  fuera de transacción, el evento se perdería silenciosamente —un fallo difícil de depurar.
- **`BEFORE_COMMIT`, no `AFTER_COMMIT`.** Solo `BEFORE_COMMIT` mantiene la escritura del Outbox
  en la misma transacción. `AFTER_COMMIT` rompería la atomicidad del patrón.
- **Una excepción en un listener `BEFORE_COMMIT` revierte la transacción.** Es lo que queremos:
  si no se puede escribir el Outbox, tampoco se confirma el cambio de estado. Coherente con 4.7.
- **Escrituras y `BEFORE_COMMIT`.** Persistir nuevas entidades durante `BEFORE_COMMIT` funciona
  con JPA, pero es un punto delicado del ciclo de vida de la sesión de Hibernate; conviene
  limitarse a `save` simples (como la fila del Outbox) y no a operaciones complejas.

---

## 5. PipelinR vs. `@TransactionalEventListener`: cuándo cada uno

| | **PipelinR (lo que implementamos)** | **`@TransactionalEventListener`** |
|---|---|---|
| **Quién despacha** | nosotros, explícito en `save` | Spring, al publicar el evento |
| **Enganche a la transacción** | implícito: corremos *dentro* del `save`, que está en la tx | explícito: `phase = BEFORE_COMMIT` |
| **Dependencia** | librería externa (`net.sizovs:pipelinr`) | nativo de Spring, sin dependencias |
| **Acoplamiento del dominio** | `DomainEvent implements Notification` (marker de PipelinR) | el dominio no implementa nada de Spring |
| **Visibilidad para enseñar** | todo el flujo en un método | el "quién escucha" queda desacoplado, menos lineal |
| **Reutiliza el mismo mediador que los comandos** | sí (si usaras PipelinR también para comandos) | no, es solo para eventos |

Ninguna es "la correcta". En el curso elegimos **PipelinR** porque el flujo queda a la vista en un
único método y usa el mismo patrón mediador que ya conocíamos para comandos. En un proyecto que
no quiera meter una dependencia en el dominio, o que prefiera apoyarse solo en Spring,
`@TransactionalEventListener(BEFORE_COMMIT)` es una alternativa perfectamente válida —y muy común.

---

## Puntos clave

- La alternativa nativa de Spring al despacho con PipelinR es **publicar `ApplicationEvent`** y
  escucharlos con **`@TransactionalEventListener`**.
- Para el Outbox hay que usar la fase **`BEFORE_COMMIT`**: mantiene la fila del Outbox en la
  **misma transacción** que el cambio de estado → **un solo commit atómico** (igual que en 4.7).
- El **punto de emisión no cambia**: los eventos se siguen recogiendo del agregado en el `save`
  del repositorio; el listener solo cambia *quién reacciona*, no *de dónde salen*.
- Ventaja del listener: **el dominio no depende de PipelinR**. Desventaja: el flujo queda menos
  lineal y solo sirve para eventos (no reaprovecha el mediador de comandos).
- `@TransactionalEventListener` **se ignora sin transacción activa** por defecto: publica siempre
  dentro del `@Transactional` del caso de uso.
